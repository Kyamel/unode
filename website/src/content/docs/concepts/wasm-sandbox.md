---
title: WASM Sandbox
description: Why Unode runs plugins in WebAssembly, how host functions work, and the memory protocol.
---

Plugins run as WebAssembly modules. WASM linear memory is isolated by the
runtime: a plugin cannot read or write host memory without an explicit host
function that grants access. This is enforced at the machine level, not by
trusting the plugin.

This is strictly stronger than iframe sandboxing (which only gates DOM access)
and stronger than Worker permissions (which gate syscalls but not in-process
memory).

## Plugin lifecycle in the sandbox

```text
1. Host reads plugin.wasm
2. Host validates the manifest (WASM export or sidecar JSON)
3. Host loads the PermissionProfile from storage
4. Host checks: are all required permissions granted?
   -> No: reject, do not instantiate
5. Host builds a host-function set filtered by the PermissionProfile
6. Host instantiates the module with only those host functions
7. Optional init() export runs (one-time setup)
8. Per route match: render -> screen JSON
9. Per action:      dispatch -> side effects via host calls
10. On unmount: the host drops (or pools) the WASM instance
```

## Host functions

Host functions are the **only** way a plugin touches anything outside its own
memory, and each is gated by a permission check before it runs.

Core host functions every renderer implements:

```text
// State
state_get(path) -> value
state_set(path, value)
state_get_snapshot() -> json

// Navigation
navigate(to, mode)
navigate_back()

// HTTP (requires "http.fetch")
http_fetch(url, opts) -> response

// Storage (requires "storage.*")
storage_get(scope, key) -> value
storage_set(scope, key, value)

// Events (requires "events.*")
events_emit(type, payload)
events_on(type) -> subscription_id

// Locale
locale_get() -> BCP 47 string, e.g. "pt-BR"
```

Domain host functions are registered by the app bridge, each with its required
permission. A function is **only injected** into the module if its permission
was granted -- see [Permissions](/concepts/permissions/).

## Memory protocol

WASM modules use linear memory. Strings and JSON are passed as `(ptr, len)`
pairs: the host reads from WASM memory at the given offset and length. Responses
use a simple allocation protocol generated by `unode-plugin-sdk` -- the `export_plugin!`
macro emits `unode_alloc` and `unode_dealloc`. The host calls `unode_alloc`,
writes JSON into WASM memory, and passes `(ptr, len)` to the export it invokes.

## Sandboxing per host

**Web** -- plugins are instantiated via `WebAssembly.instantiate()`; the browser's
WASM runtime enforces memory isolation. Host functions are JavaScript closures
owned by the trusted web runtime/bridge and check permissions before executing:

```typescript
const imports = {
  unode: {
    http_fetch: (urlPtr, urlLen, optsPtr, optsLen) => {
      const url = readString(memory, urlPtr, urlLen);
      guard.assertOrigin(url);   // throws if origin not approved
      return fetch(url).then(/* ... */);
    },
    catalog_get_work: (idPtr, idLen) => {
      guard.assert("catalog.read");  // throws if not granted
      /* ... */
    },
  },
};

const instance = await WebAssembly.instantiate(wasmBytes, imports);
```

**TUI** -- plugins run inside Wasmtime, whose sandbox prevents any syscall or
memory access outside the module. Host functions are Rust closures registered by
`unode-tui-runtime`, each asserting its permission before doing work. The
Wasmtime instance can also be denied network capability at the OS level, so no
host function can reach the network regardless of what the Rust code does.

## Plugin-to-plugin communication

Plugins never talk directly. All inter-plugin messaging goes through the host
event bus, gated by `events.read` / `events.write` permissions:

```text
Plugin A:  ctx.events.emit("screen.refresh", { pathname: "/browse" })
             -> events_emit(...) -> host bus broadcasts
Plugin B:  ctx.events.on("screen.refresh", handler)
             -> events_on(...) -> host dispatches to Plugin B
```

The host controls which plugins may subscribe to which event types.
