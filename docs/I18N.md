# unode i18n and Locale

## Design decision

Locale awareness is split between unode core and the app bridge:

- **unode core** defines the locale contract -- how the current locale is exposed
  to plugins.
- **The app bridge** implements the locale provider -- it knows the app's current
  locale setting and exposes it via a host function.
- **Plugins own their translations** -- they ship their own catalogs and resolve
  them through SDK helpers during load and render.

This split exists because an app might be monolingual (no locale switching needed)
or might have complex locale switching logic tied to user preferences, URL params,
or account settings. unode should not own that complexity.

---

## The locale contract in unode core

unode defines one host function and one plugin API:

```rust
// Host function -- every renderer must implement this
// Returns a BCP 47 locale string: "pt-BR", "en-US", "ja"
locale_get() -> String
```

The plugin SDK exposes this as:

```rust
impl PluginContext {
    pub fn locale(&self) -> String {
        // calls locale_get() host function
    }
}
```

That is the entirety of unode's locale involvement. The core does not know
what locales an app supports, how switching works, or what happens when the
locale changes.

---

## Plugin-owned translations

Plugins ship their own JSON message catalogs and load/register them through the
plugin SDK:

```rust
ctx.i18n.register(json!({
    "en": {
        "screen_title": "Browse manga",
        "load_more": "Load more"
    },
    "pt-br": {
        "screen_title": "Explorar mangás",
        "load_more": "Carregar mais"
    }
}));
```

The SDK/plugin-side i18n helper handles:
- Catalog storage per plugin id
- Locale matching with fallback (`pt-BR` -> `pt` -> `en`)
- Simple `{variable}` interpolation
- Key-not-found fallback (returns the key itself)

```rust
let t = ctx.i18n.translator();
let title = t.t("screen_title", None, None); // "Browse manga"
let more  = t.t("load_more",    None, None); // "Load more"
```

## What the app bridge provides

The bridge implements the `locale_get` host function based on the app's own
locale state:

```rust
// In the app bridge (Rust)
linker.func_wrap("unode", "locale_get", |mut caller: Caller<'_, AppState>| {
    let locale = caller.data().user_preferences.current_locale.clone();
    write_string(&mut caller, &locale)
})?;
```

The bridge may also expose higher-level locale helpers as part of the domain
API if the app needs them, but this is optional and app-specific.

---

## Locale changes

When the user changes locale, the app triggers a full re-render of the active
screen. The host calls `plugin_render()` again with the updated locale available
via `locale_get()`. This is correct because locale changes typically affect
most visible strings, making a full re-render cheaper than tracking
per-expression locale dependencies.

The previous TypeScript implementation had i18n as a core runtime concern with
all plugins registering catalogs through a shared host registry. That created an
implicit coupling: the runtime knew about translation catalogs, which are
plugin-authored content. The current model keeps the locale source in the bridge
and the catalogs in plugin/SDK space.

---

## Lazy labels in registries

For plugin-registered navigation items and commands, translations must be
resolved lazily -- at query time, not at registration time. This is because
catalogs may be registered after the navigation items are declared.

The plugin SDK handles this with `msg(key)`:

```rust
// Registration time -- catalog may not be registered yet
ctx.navigation.register(NavigationItem {
    id: "catalog.browse.nav".into(),
    label: msg("nav_label"),  // lazy -- resolved when queried
    to: "/app/mangas/browse".into(),
    ..Default::default()
});

// Later in activation
ctx.i18n.register(catalogs);  // catalogs now available
```

When the host queries navigation items, the registry resolves lazy labels with
the active locale and the plugin-owned catalog metadata available to the SDK.

## What unode core owns vs what it does not

| Concern | Owner |
|---|---|
| Catalog storage and lookup | Plugin SDK / plugin-owned data |
| Locale fallback logic | Plugin SDK |
| Simple interpolation | Plugin SDK |
| `msg()` lazy label helper | Plugin SDK |
| Current locale value | App bridge |
| Locale switching UI | App shell (not unode) |
| Locale persistence | App bridge |
| Plural rules, complex formatting | Plugin (use any Rust i18n library) |
