# unode I18N

## Decision

I18n is part of the `unode` core.

It should not depend on:

- the host app's UI framework
- an app-specific bridge
- a renderer-specific translation layer

## Goal

Plugin authoring should stay simple.

For the common case, a plugin author should be able to:

1. ship JSON translation catalogs
2. register them with the core
3. call core i18n helpers during authoring and actions

This should feel easier than wiring host-specific i18n by hand.

## Recommended model

### Catalog format

The default format should be JSON catalogs.

Example:

```json
{
  "title": "Hot manga",
  "action.load_more": "Load more"
}
```

This gives the core a simple portable baseline.

### Registration

Plugins register their catalogs with the core runtime or core plugin context.

Conceptually:

- plugin id
- locale key
- JSON catalog

The exact API surface can be finalized later, but the contract should be core-owned.

### Usage

The core should expose helpers such as:

- current locale access
- translator lookup for the plugin's registered catalogs
- fallback resolution
- interpolation for simple values

The emitted AST should still remain serializable. Translation resolution should produce plain strings or expression-ready values, not embed framework-specific i18n objects into the AST.

### Authoring convention

Inside plugin `load()` and `render()` functions, prefer:

```ts
const t = ctx.i18n.t;
t('screen_title');
```

instead of calling `ctx.i18n.t('screen_title')` inline every time.

This convention keeps the runtime JSON model intact while making it easier to
use editor tooling such as inlang Sherlock with a `t()` matcher.

### Tooling direction

The `unode` core intentionally keeps i18n runtime-based:

- plugin-owned JSON catalogs
- locale fallback in the core
- simple interpolation at runtime

This is different from compile-time systems such as Paraglide, but it fits
dynamic plugins better.

To improve DX without changing the runtime model, plugin folders can opt into a
local inlang project that points at `./messages/{locale}.json` and matches
`t('key')` calls in the plugin source.

## What the renderer owns

The renderer still owns:

- current locale source
- locale updates
- rerender or refresh behavior when locale changes

The core owns:

- plugin catalog registration
- translation helper behavior
- fallback semantics

## Why this belongs in the core

This decision improves:

- plugin portability
- authoring ergonomics
- consistency across apps
- reduced dependency on host-specific i18n systems

It also fits the broader `unode` philosophy:

- plugin authors work at the intent level
- the core provides common ergonomics
- renderers and bridges stay focused on their own responsibilities

## What stays out of scope for now

The core does not need a maximal i18n feature set on day one.

The first version can stay intentionally small:

- JSON catalogs
- locale fallback
- simple interpolation

More advanced formatting can be added later if truly needed.
